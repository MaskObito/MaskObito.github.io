OGNL:

OGNL 是 Struts 2 默认使用的强大表达式语言。它不仅可以访问对象的属性，还可以调用 Java 的静态方法和构造方法。正是这种“强大”的特性，如果用户输入被直接当作 OGNL 执行，就为代码执行打开了大门。

漏洞分析与利用：

web279-S2-001远程命令执行

S2-001是Apache Struts2框架的一个远程代码执行漏洞，它在用户输入数据时，没有对数据进行恰当的验证和过滤，导致恶意代码可以被执行。

漏洞原理：

Struts2框架在处理某些标签（例如<s:url>、<s:a>、<s:property>等）的属性值时，会使用OGNL（Object-Graph Navigation Language）表达式来解析。当这些标签的属性值直接使用了用户可控的输入时，攻击者就可以构造恶意的OGNL表达式，从而执行任意代码。

具体来说，当开发者在JSP页面中使用了%{...}这样的OGNL表达式，并且其中的内容来源于用户输入时，漏洞就会发生。攻击者可以在请求参数中插入恶意的OGNL表达式，例如%{1+1}，服务器在解析JSP页面时会执行这个表达式。

影响范围：

Struts 2.0.0 到 Struts 2.0.8 版本都受到此漏洞影响。

攻击方式：

攻击者通常通过HTTP请求参数来注入恶意OGNL表达式。例如，如果一个URL参数name的值被直接用于OGNL表达式求值，攻击者可以构造请求，

这个OGNL表达式会禁用一些安全限制，然后执行任意Java代码，例如读取服务器上的文件内容。

修复方案：

  • 升级Struts2版本： 尽快升级到不受S2-001漏洞影响的Struts2版本（2.0.9或更高版本）。
  • 输入验证和过滤： 对所有用户输入进行严格的验证和过滤，避免将用户输入直接用于OGNL表达式求值。
  • 避免使用%{...}形式的OGNL表达式直接处理用户输入。

简单来说，S2-001漏洞是因为Struts2在处理某些标签的属性值时，对用户输入的OGNL表达式没有做好安全限制，导致攻击者可以利用OGNL表达式执行恶意代码。修复方法就是升级框架版本，并对用户输入进行严格的安全检查。

web280-S2-005远程命令执行

Struts2-005 漏洞是对 Struts2-003 漏洞的绕过补丁的再绕过。其核心原理是 OGNL 表达式注入。

  • OGNL： 是 Struts2 框架中使用的一种强大的表达式语言，它可以访问和操作 Java 对象的数据，甚至执行 Java 代码。
  • 漏洞触发点： 在参数解析过程中，攻击者可以将恶意的 OGNL 表达式作为请求参数的一部分提交。
  • 漏洞原因： 由于 Struts2 对用户输入过滤不严，这些恶意的 OGNL 表达式会被框架直接解析执行。

web281-S2-007远程命令执行

age来自于用户输入，当配置了验证规则 -validation.xml 时, 传递一个非整数给id导致错误，后端默认会将用户提交的表单值通过字符串拼接，然后执行一次 OGNL表达式并返回，从而导致了漏洞。

影响版本：Struts2 2.0.0 - Struts2 2.2.3

流程如下：

  1. 参数转换： Struts2 框架尝试将用户输入的字符串 abc' + (恶意OGNL) + ' 转换为 Integer 类型。显然，这失败了，会产生一个转换错误。框架会记录这个错误，并将原始的用户输入字符串（即 abc' + (恶意OGNL) + '）继续保存在值栈中，等待后续处理。

  2. 调用验证框架： 由于转换失败，流程会继续执行验证框架。验证框架会检查 age 字段的规则。

  3. 构建错误消息（漏洞发生！）： 验证框架发现 age 字段不仅转换失败，而且也无法通过后面的 int 验证器（因为根本不是数字）。它需要生成错误消息。

    问题就在这里：框架在生成错误消息时，会将失败字段的值（即我们输入的恶意字符串）直接拼接到错误消息中！

    它生成的错误消息模板大概是这样的：

    “Invalid field value for field “age”: ${value}”

    这里的 ${value} 会被替换成当前 age 的值，也就是 abc' + (恶意OGNL) + '。

  4. OGNL 表达式被二次解析： 当 Struts2 准备将这条错误消息渲染到视图（如 JSP 页面）时，它会再次对消息中的内容进行 OGNL 解析。因为它认为错误消息是“安全”的、由框架自己生成的。

    然而，它没有料到，消息中嵌入的 value 是来自用户输入的、未经充分处理的原始数据。当解析到 ${value} 时，它实际上开始解析以下内容。

web282-S2-008多姿势rce

在 S2-008 中，攻击面更大，主要体现在：

  1. Cookie 注入点： 漏洞允许通过 HTTP Cookie 头来注入 OGNL 表达式。这是 008./cookie.jsp 这个 payload 的关键。
  2. 参数名注入点： 除了参数值，参数名本身也成为了注入点。
  3. 默认参数拦截器： 只要使用了默认的参数拦截器，就可能受到攻击，使得很多默认配置的应用直接中招。

简单来说，S2-008 漏洞意味着，攻击者可以将恶意的 OGNL 代码放在 HTTP 请求的 Cookie 值里，提交给服务器后，Struts2 框架会错误地解析并执行它。

devmode:

struts2 应用开启 devMode 模式后会有多个调试接口能够直接查看对象信息或直接执行命令，这种情况在生产环境中几乎不可能存在，专门丢了一个开启了 debug 模式的应用到服务器上作为后门也是有可能的.

当 struts.devMode=true 时，Struts2 会暴露一个特殊的调试功能。攻击者可以通过访问特定的 URL 参数来直接执行 OGNL 表达式。

debug=command： 这个参数告诉 Struts2 的调试器：“我要执行一个命令（OGNL表达式）”。

表达式：

  1. #_memberAccess["allowStaticMethodAccess"]=true
    • 目的： 允许在 OGNL 表达式中调用静态方法。这是执行 @java.lang.Runtime@getRuntime() 所必需的。
  2. #foo=new java.lang.Boolean("false")
    • 目的： 创建一个值为 false 的 Boolean 对象，并放入变量 foo 中。
  3. #context["xwork.MethodAccessor.denyMethodExecution"]=#foo
    • 目的： 这是关键的一步。 它修改了 Struts2 上下文中的一个安全开关 denyMethodExecution（拒绝方法执行），将其设置为 false。这意味着允许执行危险的方法（如 exec()）。这一步在某些版本的沙盒限制下是必要的。
  4. @org.apache.commons.io.IOUtils@toString(...)
    • 目的： 调用 Apache Commons IO 库中的 IOUtils.toString 静态方法。这个方法用于将一个输入流（InputStream）的内容转换为字符串。
  5. @java.lang.Runtime@getRuntime().exec('env').getInputStream()
    • 目的： 这是嵌套在 IOUtils.toString 内部的命令。
    • @java.lang.Runtime@getRuntime()： 获取 Java 运行时环境。
    • .exec('env')： 执行系统命令 env。
    • .getInputStream()： 获取命令执行过程的输出流。

整个表达式的逻辑链：

  1. 放宽安全限制（允许静态方法、允许方法执行）。
  2. 执行 env 命令。
  3. 获取命令执行的输出流。
  4. 使用 IOUtils.toString 将输出流的内容读取成一个字符串。
  5. 这个字符串最终会被 Struts2 的调试页面直接显示出来，从而完成命令执行回显。

web283-S2-009

S2-009 的核心思路是：既然严格检查了“参数名”，那就不把Payload放在参数名里了，把它放在“参数值”里，然后想办法让这个值被执行。

漏洞原理

  1. 一个关键特性： 在 Struts2 中，如果一个 Action 定义了一个公共的 setter 方法，例如 setExample(String value)，那么它就会接受一个名为 example 的参数。这个参数的值会被 Struts2 通过 OGNL 设置到 Action 的对应属性中。重要的是，这个 example 参数及其值会被存入 OGNL 的上下文环境中。

  2. 攻击链构造：

    • 第一步：放置Payload。攻击者将恶意的 OGNL 表达式作为某个参数的值提交。由于 Struts2 只对参数名进行严格检查，而对参数值的检查宽松得多，这个 Payload 可以顺利通过 ParametersInterceptor 拦截器的检查。

      text

例如，Payload 可能是获取 Runtime 并执行命令的表达式。

第二步：触发执行。现在，Payload 已经作为字符串存储在 OGNL 上下文中的一个变量（比如 example）里了。如何执行它？攻击者通过一个精心构造的参数名来触发。

text

      • 这个参数名 (example)(‘xxx’) 本身也是一个 OGNL 表达式。
      • OGNL 对其的解析过程是：
        1. 首先，计算 example。这正好找到了我们第一步提交的、存储在上下文中的那个参数，它的值就是我们放置的 <OGNL_Payload_Here>。
        2. 然后，OGNL 会将这个值（我们的恶意表达式）作为一个新的 OGNL 表达式进行求值。
        3. (‘xxx’) 是这个表达式求值时的参数（虽然我们的Payload可能并不需要这个参数，但语法上需要它来构成一个“方法调用”的形式）。
      • 最终，藏在参数值里的恶意 OGNL 表达式就被成功执行了。

攻击者会构造如下 URL：

  1. name=%40java.lang.Runtime%40getRuntime%28%29.exec%28%27calc%27%29： 将恶意 OGNL 表达式 URL 编码后作为 name 参数的值提交。它被存入 OGNL 上下文。

web286-S2-015

Struts 2 在处理特定类型的 URL 请求时，未能正确区分数据和代码。它将本应视为普通字符串的用户输入（如 URL 路径中的 Action 名），错误地当成了 OGNL 表达式进行解析和执行，从而让攻击者可以注入恶意代码，在服务器上执行任意命令。

两大触发场景

  1. 动态方法调用启用时：
    当 struts.enable.DynamicMethodInvocation = true 时，攻击者可以通过 actionName!恶意OGNL表达式.action 的形式触发漏洞。
  2. 通配符配置结果路径时：
    当 struts.xml 中配置了 <action name="*"> 和 <result>/{1}.jsp</result> 时，攻击者可以通过访问 /%{恶意OGNL表达式}.action 来触发漏洞，因为 {1} 的内容会被执行。
